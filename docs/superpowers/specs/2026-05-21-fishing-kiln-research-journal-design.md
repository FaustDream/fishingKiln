# fishingKiln 研究杂志站增强设计

## 1. 目标

在现有“渔窑手工青花”静态内容站基础上，继续保留品牌气质，但把网站升级为一个更完整的 `青花研究杂志站`。

本次设计的目标不是把网站改成电商、后台系统或重交互单页应用，而是让用户可以围绕以下主线持续阅读：

1. 青花历史与景德镇脉络
2. 工艺步骤与术语理解
3. 器物分类与研究专题之间的双向跳转
4. 从首页导览进入连续阅读，而不是停留在轻量摘要层

## 2. 已确认设计结论

### 2.1 网站定位

- 类型：静态多页研究型内容站
- 核心语气：青花品牌站 + 小型研究馆 + 编辑式阅读体验
- 技术边界：不引入前端框架，不增加后端，不改成 CMS，不增加交易流程

### 2.2 首页主体验

- 总方向：`研究杂志型`
- 首页首屏主线：`青花历史与景德镇脉络`
- 首页首屏布局方案：`A. 编年时间线型`

这意味着首页不再只是品牌封面，而是用户进入整站后第一篇可读的“总导览文章”。

### 2.3 内容组织原则

- 历史、工艺、器物、专题必须能相互跳转
- 内容的丰富主要来自结构化数据与前端渲染，不来自后端系统
- 交互必须服务阅读，而不是为了做出复杂控件
- 所有新增模块都应保持当前站点的克制、安静、青花水墨气质

## 3. 信息架构

### 3.1 首页：历史入口页

首页升级为完整导览页，承担以下职责：

1. 用时间线建立“青花如何形成并传播”的总脉络
2. 用景德镇背景面板解释窑口、原料、御窑与海贸的重要性
3. 向下连接研究专题索引、工艺步骤、器物导览和术语卡
4. 让用户从首页即可进入更深的专题阅读

首页结构建议按以下顺序组织：

1. 编年时间线 Hero
2. 景德镇辅助面板
3. 工艺步骤导览
4. 研究索引区
5. 器物导览区
6. 术语 / 脉络卡区

### 3.2 研究专题详情页

当前 `资料札记` 使用卡片 + 弹窗摘要，阅读深度不足。本次应新增真正的专题详情能力。

建议新增：

- `research.html`

建议通过 URL 参数读取具体条目，例如：

- `research.html?id=yuan-origins`

专题详情页应包含：

1. 专题标题
2. 导语
3. 关键事实列表
4. 相关时代或主题标签
5. 相关器物或分类入口
6. 来源链接
7. 返回首页索引或继续深读的路径

### 3.3 研究索引层

首页中段新增一层“研究索引”，作用是把现有内容从线性阅读扩展为主题浏览。

建议支持的索引维度：

- 时代
- 用途
- 器型
- 纹样主题
- 工艺关键词

索引结果应输出为一组研究专题卡，而不是仅修改一段文案。

### 3.4 分类页专题化

现有 `tea.html`、`tableware.html`、`coffee.html`、`vase.html`、`art.html` 保留，但内容升级为“小专题页”。

每个分类页建议包含：

1. 该器类的专题导语
2. 器型 / 用途 / 纹样专题切换
3. 对应历史节点提示
4. 相关研究专题链接
5. 与术语卡的关联提示

分类页应保持当前的多页结构，不需要改造成统一的客户端路由系统。

### 3.5 知识辅助模块

新增两类轻量知识模块：

1. `青花术语卡`
   - 示例：分水、釉下彩、御窑、外销瓷

2. `景德镇脉络卡`
   - 示例：窑口、原料、运输、海贸、作坊体系

这些模块可在首页、研究详情页和分类页复用，用于建立跨页面内容连接。

## 4. 首页首屏设计

### 4.1 编年时间线 Hero

首页首屏采用 `编年时间线型`，核心不是装饰，而是让用户第一眼就理解研究主题。

建议首屏至少包含 4 个节点：

1. 元代定型
2. 明清扩展
3. 景德镇系统
4. 全球传播

节点点击后应产生明确阅读反馈，例如：

- 跳转到首页对应下方专题模块
- 切换关联研究卡
- 高亮相关术语卡

### 4.2 景德镇辅助面板

时间线旁边增加一块固定辅助面板，作用是用简短但有信息密度的方式说明：

- 为什么景德镇重要
- 它与御窑、原料和海贸之间的关系
- 它如何影响青花成为可识别的文化对象

### 4.3 阅读与视觉关系

首屏必须先保证内容可读，再谈动效。

允许：

- 节点高亮
- 平缓切换
- 轻量悬停反馈

不建议：

- 大面积自动轮播
- 高频位移动效
- 把关键信息藏在 hover 后才出现

## 5. 关键交互设计

### 5.1 首页时间线联动

当用户点击时间线节点时：

1. 节点本身高亮
2. 首页下方对应专题卡同步强调
3. 相关术语或景德镇脉络卡被联动标记

交互的目的不是“做效果”，而是帮助用户理解内容之间的关系。

### 5.2 研究索引筛选

索引区允许用户按单一维度或轻量组合维度浏览研究专题。

筛选结果应：

- 保持卡片网格稳定
- 无结果时显示清晰的空状态
- 提供“清除筛选”入口

### 5.3 研究详情阅读

从研究卡进入详情页后，应支持：

- 查看完整专题内容
- 跳回首页索引
- 跳转到相关器物分类页
- 继续进入其他相关研究条目

### 5.4 分类页专题阅读

分类页的 tab 不只是切换说明段落，而应切换具有研究意味的专题面板，例如：

- 某类器物的典型用途
- 常见器型差异
- 常见纹样主题
- 与哪段历史脉络最相关

## 6. 内容与数据设计

### 6.1 数据组织原则

当前仓库已经把内容拆到：

- `js/content/home.js`
- `js/content/categories.js`
- `js/content/research.js`

本次应延续这一方向，把新增内容继续结构化，而不是把大量 HTML 字符串散落回页面文件。

### 6.2 建议新增或调整的数据块

建议新增以下数据能力：

1. 首页时间线数据
2. 景德镇背景面板数据
3. 研究索引标签数据
4. 研究详情数据
5. 术语卡数据
6. 分类页关联历史节点数据

若现有 `researchItems` 数据字段不足，应扩展字段，而不是另起一套不一致的数据源。

### 6.3 与旧详情数据的关系

仓库中已有旧的 [`js/detailData.js`](D:/gitHub/fishingKiln/js/detailData.js) 思路，可作为“详情阅读”历史线索参考，但本次实现应优先采用模块化、结构化、可测试的数据组织方式，而不是回到全局 `window` 注入。

## 7. 技术实现边界

### 7.1 保留现有结构

继续使用当前技术栈：

- 静态 HTML 页面壳
- 模块化 JavaScript
- 渲染函数
- 轻交互脚本
- `node --test` 测试体系

### 7.2 新增页面与能力

建议新增：

- `research.html`

建议增强：

- `js/renderers.js`
- `js/interactions.js`
- `js/main.js`
- `js/siteContent.js`

- `js/content/home.js`

必要时可新增专门的数据或渲染模块，但应保持职责清晰，不做与目标无关的抽象扩张。

### 7.3 明确不做

以下内容不在本次范围内：

## 8. 失败处理与兜底

为了保证阅读体验完整，以下失败路径必须在设计阶段就被纳入：

### 8.1 研究条目不存在

当 `research.html` 的 `id` 无效时，应显示明确的“未找到该专题”，并提供：

- 返回首页
- 返回研究索引

### 8.2 URL 缺少参数

当详情页没有 `id` 参数时，不允许显示空白页。应退回默认导读或提示用户从研究索引进入。

### 8.3 索引筛选无结果

当筛选条件没有匹配内容时，应显示空状态文案，并提供清除筛选的入口。

### 8.4 条目缺图或缺来源

当个别条目没有图片或来源时：

- 不显示破图
- 不出现空链接
- 使用文字型卡片作为降级表达

### 8.5 动效不可用

当用户设备不支持某些交互或禁用动效时，页面仍必须能完整阅读，不得把关键信息绑定在纯视觉特效上。

## 9. BDD 验收场景

本次设计要求在实现前先写外环验收场景，用于定义“什么叫完成”。

### 9.1 首页历史主线

```gherkin
Scenario: 首页首先呈现青花历史与景德镇脉络
  Given 用户进入首页
  When 页面完成渲染
  Then 用户首先能看到编年时间线
  And 用户能看到景德镇相关背景说明
```

### 9.2 研究索引筛选

```gherkin
Scenario: 用户按主题筛选研究专题
  Given 用户进入首页研究索引区
  When 用户选择一个时代或工艺关键词
  Then 页面展示对应的研究专题卡
  And 不相关专题不会继续作为主结果显示
```

### 9.3 研究卡进入详情页

```gherkin
Scenario: 用户从研究卡进入连续阅读
  Given 用户看到一张研究专题卡
  When 用户打开该专题
  Then 页面显示完整专题内容
  And 页面提供返回索引或继续深读的路径
```

### 9.4 分类页专题化阅读

```gherkin
Scenario: 用户在分类页继续阅读相关专题
  Given 用户进入任一器物分类页
  When 用户切换专题面板
  Then 页面展示该器类的研究性内容
  And 页面继续提供相关历史节点或研究链接
```

### 9.5 异常场景兜底

```gherkin
Scenario: 研究专题不存在时给出友好反馈
  Given 用户打开一个不存在的研究专题链接
  When 页面尝试解析专题参数
  Then 页面显示未找到专题的提示
  And 页面提供返回首页或研究索引的入口
```

## 10. TDD 策略

只要进入生产代码实现，就必须先写失败测试，再写最小实现。

### 10.1 优先进行 TDD 的模块

1. 时间线节点渲染
2. 研究条目查找逻辑
3. URL 参数解析与详情页回退
4. 研究索引筛选函数
5. 术语卡与专题关联逻辑
6. 空状态与缺失数据兜底

### 10.2 单测原则

- 每个测试只验证一个行为
- 先确认失败原因正确，再写实现
- 尽量测试真实渲染与真实数据，不为方便测试而改写生产接口

## 11. 验证标准

当实现完成后，应满足以下标准：

1. 首页第一屏以历史时间线和景德镇脉络为首要信息
2. 用户可以从首页进入研究专题详情页
3. 用户可以通过研究索引筛选内容
4. 分类页不再只是浅层栏目页，而具备专题阅读能力
5. 术语卡和景德镇脉络卡在至少两个页面层级中复用
6. 异常路径存在明确兜底，不出现空白页和破碎跳转
7. BDD 外环场景有对应测试表达
8. TDD 内环覆盖关键渲染、筛选和兜底逻辑

## 12. 推荐实施结论

后续实现应围绕以下一句话执行：

`把渔窑从一个有气质的品牌内容站，升级为一个能围绕青花历史、景德镇脉络、器物分类与研究专题持续阅读的静态研究杂志站。`
